// How I approach system design

THINKING
01
Define the contract, not the code

Start with API contracts and event schemas. Every downstream decision becomes easier once data boundaries are clear.

02
Model failure, then success

What happens when the DB is down? When a message is replayed? When latency spikes? Resilience is designed, not retrofitted.

03
Scale reads before writes

CQRS, read replicas, and caching layers defer vertical scaling. Write paths deserve separate optimization.

04
Ship observable, iterate fast

Trace IDs, structured logs, and dashboards from day one. If it's dark in production, it's already broken.
